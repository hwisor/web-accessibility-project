Accessibility Benefits Everyone

Web Accessibility Project provides a quick reference for those who are new to designing for web accessibility. It is important that the web is designed and developed to provide equal access to people with disabilities. Specifically, people with disabilities that affect their vision, hearing, and movement.

There's a misconception that design must be sacrificed in order to have an accessible website. Many sites are developed without thinking of accessibility as part of the process, making them difficult or impossible to use. The same principles that make a website accessible, however, are not not just for people with disabilities. Legible typography, clutter-free layouts, and organized content are just some of the ways to make a good website experience that benefits all users.

This web project is a break down of the Web Content Accessibility Guidelines, (WCAG) that designers and developers must follow in order to have a compliant and accessible site. It covers the most basic, but important, aspects of accessibility design. This page is not an in-depth checklist of accessibility standards, but an introduction to those who are new to accessibility design for the web.

The task of creating a website that is accessible to all may seem like a daunting task. Web Accessibility Project hopes to provide the starting point for your web accessibility education. It’s important to develop an ongoing practice of reviewing and testing your site to identify accessibility issues.

Accessibility is essential for designers that want to create visually-appealing websites, but also provide equal access to everyone.

Design Considerations

Semantics and Structure

HTML tags are used properly and in the correct DOM order to be programmatically determined.

Alternative Text

Non-text content has a text alternative. This includes captions and transcripts for multimedia content.

Keyboard Navigation

The functionality of content is available using only the keyboard and focusable elements have tab focus.

Programmatic Labels

Custom controls have the correct name/label, role, and value (either with HTML or with ARIA).

Color and Contrast

Elements have sufficient contrast, and color is not the only visual means of conveying information.

Testing

Using automated tools and assistive technology, like screen readers, to check for accessibility.

Semantics and Structure

Semantics, or semantic markup, refers to the use of HTML tags to convey the meaning of the content contained within them. Semantic elements use words to represent what that element contains, such as information, structure, and relationships.

For example, headings are one of the most important semantic elements of any web page. They represent the titles and subtitles of a page. They provide a clear structure of a document, which allows all users to easily scan through the page content. There are six levels of headings: < h1 >,< h2 >, < h3 >, < h4 >, < h5 >, and < h6 > . The beginning of the main content should always start with < h1 > and there should only be one < h1 > element. Headings should not skip hierarchical levels. Text that does not act as a heading visually or structurally should not be marked as a heading.

Semantic elements (landmarks, headings, lists, links, images, etc.) are typically read aloud by screen readers. When screen reader users navigate through the content, the order that they hear the content is determined by the order in the document object model (DOM). It’s important then that the reading order of the semantic markup is logical and intuitive for screen reader users.

Alternative Text

Non-text content, such as images, graphics and videos, are not visible for the blind or the visually impaired. All non-text content should provide a text alternative that represents the same information as the visual and multimedia content that can be interpreted by assistive technologies.

Images and graphics must have descriptive alternative text. This can be with the alt attribute value (alt=), title attribute value (title=), aria-label property, or aria-labelledby property.

Good Example:
< img src="cute-black-kitten.jpg" alt="A black kitten" >

Bad Example:
< img src="IMG_584321.jpg" >

Similarly, audio content is not perceivable to those who are deaf or hard of hearing. Webpages should also provide captions and transcripts for their multimedia content.

Learn more about Alternative Text

Keyboard Navigation

Many people use only the keyboard to navigate web pages. Therefore, all functionality of the content should be available through a keyboard.

Keyboard users typically navigate through a web page using the Tab key. Links, buttons, and interactive controls must be focusable for keyboard users to tab through. The keyboard focus should never be locked or trapped on a page element.

The order of focusable items (buttons, links, form elements, custom controls, etc.) also needs to be logical and intuitive for keyboard users to easily navigate the content.

Programmatic Labels

Every form input and control needs a label for screen readers and other assistive technologies to communicate the name, role, and value to the user.

There are multiple ways of providing labels. The best way is to use the HTML < label > element because it works and is already accessible. Another way is by using Accessible Rich Internet Applications (ARIA), a set of roles and attributes that provide semantic meaning to the content. ARIA is best for more complex and dynamic interactions that can’t be communicated using simple HTML elements.

ARIA allows you to communicate the following to screen readers:

  • labels or names for items (aria-label or aria-labelledby)
  • roles, role="navigation", role="main", etc.
  • states of dynamic/interactive components ( aria-selected="true", aria-expanded="true", aria-hidden="true")
  • properties of items (aria-haspopup="true")
  • relationships between items (aria-owns, aria-controls)
  • live announcements in real-time that are passed on to screen readers.

Color and Contrast

Color is great for enhancing the content and design of a webpage, but it should not be the only visual means of conveying information. There should be other visual indicators besides color so that the visual content is perceivable and understandable by people of differing abilities, including people who are color blind or have low vision.

Color contrast is also important for those who need to magnify the screen, or who need high contrast colors. The color contrast between text and background should be at least 4.5:1 for small text or 3:1 for large text so the content is easier to read.

Testing

Testing for accessibility should be conducted in the early and main stages of the development process in order to catch accessibility issues before they reach production.

The best practice is to start by conducting automated tests. There are a number of software programs and tools that can be used to automatically scan pages on a website to identify accessibility problems, such as axe DevTools extension and Firefox Web Developer Tools.

Manual testing then needs to be done in order to validate automated tests and discover issues that automated tests didn’t find. The main methodology for manual testing is the use of screen readers.

There are a number of screen readers to test with:

Desktop screen readers:

  • JAWS with Chrome
  • NVDA with Firefox
  • NVDA with Chrome
  • JAWS with IE
  • VoiceOver with Safari (macOS)
  • Narrator with Edge

Mobile screen readers:

  • VoiceOver with Safari (iOS)
  • TalkBack with Chrome (Android)

After conducting both automated and manual testing, the findings and issues from the tests should be documented in an accessibility report to be shared with designers and developers.

Further Information

Website accessibility is a legal matter in many countries, with accessibility non-compliance amounting to discrimination. Web Content Accessibility Guidelines (WCAG), published by the World Wide Web Consortium (W3C), define the principles, guidelines, success criteria, and techniques needed for accessibility.

Review their detailed methodology to see how your website adheres to accessibility standards.
